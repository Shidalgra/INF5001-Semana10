Es totalmente normal que se sienta extraño pasar de un mundo tan estructurado 
como el de SQL (donde todo son tablas 📊, filas, columnas y claves foráneas) 
a uno NoSQL como MongoDB.

Para que tu mente haga el puente rápido entre lo que ya conoces y lo nuevo, 
mira esta equivalencia básica:

*   Base de datos 🗄️ ➔ Sigue llamándose Base de datos.

*   Tabla 📋 ➔ En MongoDB se llama Colección (Collection).

*   Fila / Registro 📑 ➔ En MongoDB se llama Documento (Document) y se 
*   guarda en un formato muy parecido a JSON.

La gran diferencia es que en MongoDB no hay "tablas relacionales" estrictas 
ni esquemas fijos obligatorios por defecto; cada documento puede tener campos 
diferentes si así se requiere.

*   ¡Se preguntarán!!! 

¿tengo que crear una BD como en SQL?

En MongoDB no necesitas crear la base de datos de forma explícita antes de usarla.

En el mundo SQL, estás acostumbrado a tener que ejecutar obligatoriamente un CREATE 
DATABASE y luego un CREATE TABLE con todas sus columnas bien definidas antes de poder 
insertar una sola fila. Si no lo haces, el gestor te lanza un error.

MongoDB funciona con un enfoque llamado creación implícita o bajo demanda (lazy creation):

Al vuelo: Cuando tú envías un documento a una base de datos o a una colección (tabla) que aún
no existe, MongoDB no falla. Simplemente revisa su sistema, nota que no existe, la crea en 
ese mismo milisegundo y guarda la información de inmediato. 

En nuestro proyecto: Por eso, cuando configuramos la URL mongodb://127.0.0.1:27017/todoDB, 
la base de datos todoDB no existía en tu computadora. Al presionar "Send" en Thunder Client,
MongoDB la creó por ti automáticamente junto con la colección de tareas.

Esta es una de las razones por las que NoSQL es tan popular para el desarrollo rápido.

Pensando en tu experiencia con la estructura firme de SQL Server, ¿qué gran ventaja o qué
peligro te imaginas que podría traer este comportamiento de "crear todo automáticamente al 
vuelo"?

Abre tu archivo index.js y agrega estos dos bloques de código justo debajo de la ruta GET

    /*--------*/
*   JAVASCRIPT    
    // Ruta para ACTUALIZAR una tarea por su ID (PUT)
    app.put('/tareas/:id', async (req, res) => {
        try {
            const tareaActualizada = await Tarea.findByIdAndUpdate(
                req.params.id, 
                { titulo: req.body.titulo },
                { new: true } // Esto hace que devuelva la tarea ya modificada
            );
            res.status(200).json(tareaActualizada);
        } catch (error) {
            res.status(500).json({ mensaje: 'Error al actualizar la tarea', error });
        }
    });

    // Ruta para ELIMINAR una tarea por su ID (DELETE)
    app.delete('/tareas/:id', async (req, res) => {
        try {
            await Tarea.findByIdAndDelete(req.params.id);
            res.status(200).json({ mensaje: 'Tarea eliminada correctamente' });
        } catch (error) {
            res.status(500).json({ mensaje: 'Error al eliminar la tarea', error });
        }
    });
    /*--------*/


Observa que ambas rutas tienen algo especial en su dirección: /:id. Esto actúa como un 
parámetro variable en Express, muy parecido a cuando usas un WHERE Id = @Id en SQL Server 
para asegurarte de no modificar o borrar toda la tabla por error.
